Internal Strategy Brief / Product UX

โครงสร้าง 4 เว็บไซต์: Direction Matrix และข้อเสนอปรับ IA

เอกสารช่วยตัดสินใจสำหรับทีมผู้บริหาร กำหนดบทบาทและขอบเขตของแต่ละเว็บให้ชัด พร้อมข้อเสนอปรับโครงสร้างเมนูแบบเทียบก่อนและหลัง อิงจากโครงสร้างจริงของเว็บ ณ เดือนมิถุนายน 2026

SURFACES: www / blog / online / classroom เรื่องที่ต้องตัดสินใจ: 3 เรื่อง VERSION: 1.3 (12 มิ.ย. 2026)

สรุปสำหรับผู้บริหาร

อ่านจบใน 3 นาที

การแบ่ง 4 เว็บตาม funnel เดินมาถูกทางแล้ว แต่ปัญหาคือขอบเขตของแต่ละเว็บยังทับซ้อนกัน และเมนูบวมเกินบทบาทของตัวเอง ผลคือ user หลงทางเวลาข้ามเว็บ ส่วนทีมภายในก็ไม่มีเกณฑ์ว่าของใหม่ควรไปอยู่เว็บไหน เอกสารนี้ขอให้ที่ประชุมตัดสินใจ 3 เรื่อง

Finding 01 / Structure

มี 4 เว็บ แต่ยังไม่ใช่เว็บละหน้าที่

โปรโมชันกับรีวิวไปอยู่บนเว็บแบรนด์ ทั้งที่เป็นของไว้ปิดการขาย ส่วนเมนูคอร์สบนเว็บขายลึกถึง 5 คอลัมน์ ราว 40 ลิงก์ ซึ่งใช้บนมือถือลำบากมาก ทั้งที่มือถือคืออุปกรณ์หลักของนักเรียน

Finding 02 / Critical Risk

blog subdomain ค้างอยู่ครึ่งทาง

blog.* ถูกสร้างแล้วแต่ยังปิดไม่ให้ Google เก็บ (noindex) ข้างในมีโพสต์ทดสอบและรูปที่ชี้ localhost ค้างอยู่ ขณะที่บทความ SEO ตัวจริงยังอยู่บนเว็บหลักทั้งหมด ถ้าเปิดใช้โดยไม่มีแผนย้ายที่รัดกุม คะแนน SEO ที่สะสมมาจะถูกแบ่งครึ่งทันที

Finding 03 / Opportunity

เมนูแบบ "เลือกตามคณะ" คือจุดแข็งที่ควรขยาย

การจัดคอร์สตามกลุ่มคณะ เช่น สายหมอ วิศวะ บัญชี ตรงกับวิธีคิดของเด็ก ม.ปลาย มากกว่าการไล่ตามรายวิชา ควรยกขึ้นเป็นแกนหลักของเว็บขาย ไม่ใช่ซ่อนเป็นแค่คอลัมน์หนึ่งในเมนู

3 เรื่องที่ขอให้ที่ประชุมตัดสินใจ

★ แนะนำA) เก็บบทความไว้บนเว็บหลักที่ /article/ เหมือนเดิมหยุดแผน blog.* หรือเก็บไว้เป็นที่ทดสอบอย่างเดียว แล้วเอาแรงไปลงกับการจัดโครงบทความใหม่แทน
B) เปิด blog.* พร้อมแผนย้ายเต็มรูปแบบต้องทำ 301 redirect รายหน้า แก้ลิงก์ภายในทั้งหมด ใช้แรง dev และ SEO สูง และยอมรับว่า traffic อาจตกช่วง 2 ถึง 6 เดือนแรก
C) ปล่อยไว้แบบนี้เสี่ยงที่สุด เพราะตอนนี้มีระบบบทความค้างอยู่ 2 ที่ ทีม content ไม่รู้จะลงที่ไหน และมีโอกาสเผลอเปิด index ซ้ำซ้อนโดยไม่ตั้งใจ

เหตุผลสั้น ๆ: Google มอง subdomain ใกล้เคียงกับอีกเว็บหนึ่ง การย้ายบทความที่สะสม backlink มานานออกจากโดเมนหลักเท่ากับเริ่มนับหนึ่งใหม่บางส่วน ในจังหวะที่กำลังแข่ง keyword สนามสอบ ความเสี่ยงนี้ไม่คุ้ม ยกเว้นมีข้อจำกัดด้านระบบที่แยกไม่ได้จริง ๆ

หมายเหตุสำคัญ: การเลือกข้อ A ไม่ได้ผูกกับ tech stack ของ CMS เลย จะใช้ WordPress ต่อหรือสร้างระบบใหม่ก็ได้ เพราะระบบใหม่เสิร์ฟผ่าน URL เดิมที่ /article/ ได้ด้วยเทคนิค reverse proxy การวิเคราะห์เทียบทั้งสองสาย (WordPress และไม่ใช่ WordPress) อยู่ในแท็บ 05 Technical Appendix

★ แนะนำอนุมัติ Direction Matrix ในแท็บ 02 เป็นเกณฑ์กลางต่อไปของใหม่ทุกชิ้นต้องตอบได้ว่าอยู่เว็บไหน ตามตาราง Scope ที่วางไว้ ช่วยลดการถกเถียงรายครั้ง และกันเมนูบวมกลับมาอีก
แก้เฉพาะจุดที่เจ็บที่สุด เช่น mega menu กับหน้า HOFเร็วกว่า แต่ปัญหาต้นเหตุที่ว่าของใหม่ไม่รู้ไปไว้ไหนจะวนกลับมาในอีก 2 ถึง 3 ไตรมาส

สิ่งที่จะได้: เมนูหลักของเว็บแบรนด์ลดจาก 7 กลุ่มเหลือ 5 กลุ่ม เว็บขายได้เมนูแบบ mobile-first 4 ทางเข้า และทุกเว็บมีตัวชี้วัดเจ้าภาพของตัวเอง รายละเอียดอยู่ในแท็บ 03

★ แนะนำทำ SSO และให้จัดการบัญชีจบได้ในที่เดียวทุกวันนี้แค่ลืมรหัสผ่านห้องเรียน ยังต้องเด้งกลับไปทำที่เว็บขาย เป็น friction กับลูกค้าที่จ่ายเงินแล้ว ซึ่งเป็นกลุ่มที่เสียไปแล้วแพงที่สุด
คงโครงเดิม แต่ปรับข้อความและลิงก์ให้ชัดขึ้นลดความงงได้ระดับหนึ่งด้วยต้นทุนต่ำ เหมาะใช้เป็น quick win ระหว่างรอ SSO

เหตุผลสั้น ๆ: เว็บห้องเรียนคือที่ที่ลูกค้าอยู่นานที่สุด และเป็นที่ที่เขาตัดสินใจซื้อซ้ำหรือบอกต่อ friction ทุกจุดตรงนี้จึงมีต้นทุนเป็น LTV โดยตรง

Technical Appendix

CMS ใหม่ บน URL เดิม /article/
สำหรับทีม DEV ใช้หลังจากที่ประชุมเคาะเรื่องที่ 1 ข้อ A แล้ว

โจทย์คือทีมจะพัฒนา blog ขึ้นใหม่ แต่ ยังไม่ได้ตัดสินใจว่า CMS จะใช้ tech stack ไหน อาจเป็น WordPress ต่อ หรือสร้างด้วยระบบอื่นก็ได้ ส่วนนี้จึงวิเคราะห์ทั้งสองสาย พร้อมยืนยันข้อสำคัญที่สุดก่อนเลยว่า ไม่ว่าเลือก stack ไหน ก็ยังใช้ URL เดิมที่ www.smartmathpro.com/article/ ได้เสมอ ต่างกันแค่ความยากของวิธีต่อระบบ เพราะ www. ปัจจุบันอยู่บน WordPress กับ shared hosting

หลักคิดก่อนเลือก stack: การตัดสินใจ 2 ชั้นที่แยกจากกันได้

ชั้นที่ 1 เรื่อง URL (บทความอยู่ที่ /article/ บนโดเมนหลัก ตามเรื่องที่ 1 ข้อ A) กับ ชั้นที่ 2 เรื่อง CMS (สร้างด้วยอะไร) เป็นคนละเรื่องกัน เพราะชั้น routing ที่ Cloudflare ไม่สนใจว่าปลายทางเป็นระบบอะไร ที่ประชุมจึงเคาะเรื่องที่ 1 ได้เลยวันนี้ โดยไม่ต้องรอทีม dev สรุป stack และทีม dev ก็เลือก stack ได้อิสระภายหลังโดยไม่กระทบ SEO

Scenario A: blog ใหม่ยังใช้ WordPress

มี 3 ท่าย่อย เรียงจากง่ายไปยาก

★ ง่ายที่สุดA1) ใช้ WordPress ตัวเดิมบน www.ทำ /article/ เป็น category หรือ custom post type ในระบบเดิม ไม่ต้องมี proxy ไม่ต้องมี server ใหม่ SEO ปลอดภัยที่สุด เหมาะถ้าข้อจำกัดหลักคือเวลา dev ข้อแลกคือยังติด theme, plugin และ performance ของ shared hosting เดิม
A2) WordPress ตัวใหม่แยก server แล้ว proxy เข้า /article/ใช้สถาปัตยกรรม Cloudflare ด้านล่างได้เหมือนกัน WordPress รองรับการอยู่ใต้ subdirectory ผ่าน WP_HOME / WP_SITEURL แต่มีจุดเจ็บคือ plugin จำนวนมากฝัง absolute URL ใน database ต้องทำ search-replace และ wp-admin หลัง proxy ต้องตั้ง header ให้ครบ effort ใกล้เคียงทำ CMS ใหม่แต่ได้ของเดิม จึงไม่ค่อยคุ้ม
A3) Headless WordPressเก็บ WordPress ไว้เป็นหลังบ้านให้ทีม content ใช้ editor ที่คุ้นเคย แล้วให้ frontend ใหม่ (เช่น Next.js) ดึงเนื้อหาผ่าน REST API หรือ WPGraphQL มาเรนเดอร์ใต้ /article/ ได้ความเร็วและ design อิสระโดย workflow ทีม content ไม่เปลี่ยน ข้อแลกคือต้องดูแล 2 ระบบ และ flow preview บทความก่อน publish ต้องทำเพิ่มเอง

เมื่อไหร่ควรเลือกสาย A: ทีม content ผูกกับ workflow ของ WordPress มาก (เขียน จัดรูป จัด SEO ผ่าน plugin เองได้โดยไม่พึ่ง dev) และกำลัง dev มีจำกัด — A1 คือ quick win ส่วน A3 คือทางยกระดับโดยไม่ทิ้งจุดแข็งนี้

Scenario B: blog ใหม่ไม่ใช้ WordPress

มี 2 ท่าย่อยหลัก ทั้งคู่ใช้สถาปัตยกรรม Cloudflare proxy ด้านล่าง และต้องทำ checklist เรื่อง base path / canonical / noindex ให้ครบเหมือนกัน

★ แนะนำสำหรับสาย BB1) Headless CMS สำเร็จรูป + frontend frameworkเช่น Payload, Strapi, Directus หรือ Sanity คู่กับ Next.js หรือ Astro ทีม content ได้หน้าจอเขียนบทความที่ไม่ต้องแตะโค้ด ส่วน dev คุมโครงสร้างข้อมูล (pillar, tag, CTA ไปคอร์ส) ได้ตั้งแต่ระดับ schema รองรับโครง pillar ในแท็บ 03 ได้ตรง ๆ
B2) Git-based / Markdown + SSGเขียนบทความเป็นไฟล์ใน git แล้ว build เป็นหน้า static ถูกและเร็วที่สุด แต่ทีม content ต้องทำงานผ่านเครื่องมือเชิง dev ซึ่งไม่ตรงกับทีม content ปัจจุบัน เหมาะกับทีมที่คนเขียนคือ dev เอง จึงไม่แนะนำสำหรับเคสเรา

เมื่อไหร่ควรเลือกสาย B: ต้องการความเร็วหน้าเว็บและ design อิสระเต็มที่ อยากออกแบบโครงข้อมูลรองรับ pillar / lead magnet / เครื่องมือ interactive ตั้งแต่ต้น และมีกำลัง dev ดูแลระบบเองระยะยาว ข้อแลกหลักคือทีม content ต้องเรียนเครื่องมือใหม่ และฟีเจอร์ที่ WordPress ได้ฟรีจาก plugin (SEO meta, redirect manager, sitemap) ต้องให้ dev ทำเอง

เทียบเร็ว สาย A กับ สาย B

  • Effort ฝั่ง dev: A1 ต่ำสุด → A3 / B1 ปานกลาง (ต้องตั้ง proxy และระบบใหม่) → A2 สูงแบบไม่คุ้ม
  • Workflow ทีม content: A1 / A3 ไม่เปลี่ยนเลย, B1 ต้องเรียนเครื่องมือใหม่ราว 1-2 สัปดาห์, B2 ไม่เหมาะ
  • ความเสี่ยง SEO: เท่ากันทุกท่า ตราบใดที่บทความอยู่ที่ /article/ ของโดเมนหลักและทำ checklist ด้านล่างครบ ความเสี่ยงจริงอยู่ที่คุณภาพการ migrate ไม่ใช่ตัว stack
  • Performance และความอิสระของ design: A3 / B1 ดีสุด ส่วน A1 ติดเพดาน theme และ shared hosting เดิม
  • ภาระดูแลระยะยาว: A1 ดูแลที่เดียว, A3 ดูแลทั้ง WordPress และ frontend, B1 ดูแล CMS กับ frontend แต่เป็น stack เดียวที่ dev เลือกเอง

ข้อเสนอวิธีตัดสิน: ตอบคำถามเดียวก่อนว่า "ทีม content ยอมเปลี่ยนเครื่องมือเขียนบทความไหม" ถ้าไม่ยอม → เลือกสาย A (เริ่ม A1 แล้วค่อยโตเป็น A3) ถ้ายอม → B1 คุ้มกว่าระยะยาวเพราะออกแบบ schema รองรับโครง pillar ได้ตั้งแต่ต้น และที่สำคัญ อย่าให้การยังไม่ตัดสินใจเรื่อง stack ไปหน่วง Phase 1 (ปิดความเสี่ยง blog.* และยืนยัน noindex) ซึ่งทำได้ทันทีไม่ว่าจะเลือกสายไหน

กรณีเลือกสาย B (รวมถึง A2 / A3): สถาปัตยกรรม reverse proxy

ข้อจำกัดตั้งต้น

Shared hosting ไม่ให้เราเข้าไปแก้ config ของ web server ตรง ๆ จึงทำ reverse proxy ที่ฝั่ง hosting เองไม่ได้ ทางออกคือยกหน้าที่นี้ไปไว้ที่ชั้นบนสุดของ traffic แทน ซึ่งก็คือ Cloudflare ที่คั่นอยู่หน้า domain

สถาปัตยกรรมที่เสนอ

ผู้ใช้เปิด www.smartmathpro.com/... | v Cloudflare (proxy คั่นกลาง ตัดสินใจจาก path) | |-- path ขึ้นต้นด้วย /article/* | ส่งไปที่ server ใหม่ที่รัน CMS ใหม่ | (host จริงปิด index ไว้ ผู้ใช้ไม่เห็น) | |-- path อื่นทั้งหมด ส่งไปที่ shared hosting (WordPress เดิม) ผลคือ user และ Google เห็นทุกอย่างเป็นโดเมนเดียว SEO authority ไม่ถูกแบ่ง และสองระบบอยู่ร่วมกันได้

วิธีตั้งค่าฝั่ง Cloudflare ใช้ได้ทั้ง Workers (ยืดหยุ่นสุด เขียน logic ได้) หรือ Origin Rules ในแผนที่รองรับ โดยทั่วไปปริมาณ traffic ระดับ blog ยังอยู่ใน free tier ของ Workers ได้

สิ่งที่ CMS ใหม่ต้องตั้งค่าตั้งแต่วันแรก

หัวใจคือ CMS ใหม่ต้องรู้ว่าตัวเองอาศัยอยู่ที่ /article ไม่ใช่ root ถ้าไม่ตั้งแต่ต้น ลิงก์ภายในและไฟล์ asset จะชี้ผิดทั้งระบบ และแก้ทีหลังเจ็บมาก ตัวอย่างกรณีใช้ Next.js

// next.config.js module.exports = { basePath: '/article', }

framework อื่นมีแนวคิดเดียวกัน เช่น Astro ใช้ base, Nuxt ใช้ app.baseURL ส่วนระบบที่เขียนเองต้อง prefix ทุก route และ asset ให้ครบ

  • Base path เป็น /article ทุกลิงก์ภายใน ทุกรูป ทุกไฟล์ JS และ CSS ต้องอยู่ใต้ path นี้
  • Canonical tag ชี้ URL เต็มของโดเมนหลัก เช่น https://www.smartmathpro.com/article/ชื่อบทความ ห้ามชี้ host จริงของ server ใหม่เด็ดขาด
  • Sitemap รวมเป็นภาพเดียว ให้ CMS ใหม่สร้าง sitemap ของตัวเอง แล้วประกาศเพิ่มใน robots.txt หรือ submit คู่กันใน Search Console ใต้ property เดียวกัน
  • Host จริงของ server ใหม่ต้อง noindex และควรบล็อกการเข้าถึงตรงด้วย ให้รับ traffic จาก Cloudflare เท่านั้น กันปัญหา content ซ้ำสองที่
  • หน้า 404 และ redirect จัดการในระบบใหม่ให้จบ บทความที่ URL เปลี่ยน slug ต้องมี 301 ภายในระบบใหม่ ไม่ปล่อยหล่นเป็น 404
  • เช็คเรื่อง trailing slash ให้นโยบายเดียวกับ WordPress เดิม ไม่งั้นจะเกิด redirect ซ้อนหรือ URL สองแบบชี้หน้าเดียวกัน
  • อย่าลืม resource นอก HTML เช่น RSS feed, OG image, ไฟล์ PDF แจกฟรี ต้องเสิร์ฟผ่าน /article/ เหมือนกัน

แผนย้ายแบบทีละขั้น ไม่ต้อง big bang

STEP 1

ตั้งระบบคู่ขนาน

CMS ใหม่ขึ้น server ของตัวเอง ปิด index สนิท ทีม content เริ่มทดลองใช้งานจริงเพื่อเช็ค workflow ระหว่างที่ WordPress เดิมยังเสิร์ฟ /article/ ตามปกติ

STEP 2

เปิด routing เฉพาะบทความใหม่

ตั้ง rule ที่ Cloudflare ให้ slug ของบทความใหม่ชี้ไประบบใหม่ ส่วน slug เก่ายังไปที่ WordPress เหมือนเดิม ช่วงนี้สองระบบทำงานคู่กันโดย user ไม่รู้สึกอะไร

STEP 3

ทยอยย้ายบทความเก่าทีละ batch

เริ่มจากกลุ่มที่ traffic สูงเพื่อรีบได้ประโยชน์จากระบบใหม่ ย้ายแล้วเทียบ URL ให้ตรงเดิมหรือใส่ 301 ให้ครบ เฝ้า Search Console ทุก batch ราว 1 ถึง 2 สัปดาห์ก่อนไป batch ถัดไป

STEP 4

สลับ routing ทั้ง /article/ แล้วปลดของเก่า

เมื่อบทความครบแล้ว เปลี่ยน rule ให้ /article/* ทั้งหมดชี้ระบบใหม่ จากนั้นถอดส่วน blog ของ WordPress ออก เหลือไว้แค่ redirect กันลิงก์เก่าตกค้าง

คำถามที่ต้องได้คำตอบจากทีม dev ก่อนเริ่ม

1. Domain ผูก nameserver กับ Cloudflare แล้วหรือยังถ้าผูกแล้วเริ่มทำ routing ได้ทันที ถ้ายังต้องย้าย nameserver ก่อน ซึ่งใช้เวลาไม่นานแต่ควรทำนอกช่วงแคมเปญ
2. CMS ใหม่สร้างด้วย stack อะไร และรองรับ base path ดีแค่ไหนคำตอบนี้กำหนดความยากของงานทั้งหมด ควรพิสูจน์ด้วย demo หน้าเดียวที่รันใต้ /article/ ให้ดูก่อน commit โครงใหญ่
3. Hosting เดิมยอมให้ Cloudflare proxy ครอบได้ไหมshared hosting ส่วนใหญ่ไม่มีปัญหา แต่บางเจ้ามีระบบ SSL หรือ security ที่ตีกับ proxy ต้องเช็ค SSL mode ให้เป็น Full อย่างน้อย
4. ใครเป็นเจ้าของ rule ที่ Cloudflarerouting layer นี้จะกลายเป็นจุดสำคัญที่สุดของระบบ ต้องกำหนดคนดูแล สิทธิ์เข้าถึง และวิธี rollback ให้ชัดก่อนเปิดใช้จริง

Account & Token Strategy

เชื่อมกับเรื่องที่ 3 (SSO) และ P0 session หลุด

ปัจจุบัน online กับ classroom ผูก login ผ่าน LINE และ Facebook แยกกันคนละชุด บวกช่องทางเบอร์โทรกับรหัสผ่าน เท่ากับเรามีระบบบัญชี 2 ระบบที่บังเอิญใช้ social provider เดียวกัน ส่วนนี้อธิบายว่าปัญหาอยู่ตรงไหน แก้ระยะสั้นได้อย่างไร และโครงระยะยาวควรเป็นแบบไหน

พื้นฐาน Token ที่ทีมต้องเข้าใจตรงกันก่อน

ระบบ login สมัยใหม่ใช้ token สองตัวทำงานคู่กัน Access Token อายุสั้นราว 15 นาทีถึง 1 ชั่วโมง ใช้ยืนยันตัวตนทุก request ที่ให้หมดอายุเร็วก็เพื่อว่าถ้าโดนขโมยจะใช้ได้ไม่นาน ส่วน Refresh Token อายุยาวเป็นหลักสัปดาห์ถึงหลักเดือน เก็บไว้เงียบ ๆ พอ access token หมด ระบบใช้ตัวนี้ขอใบใหม่อัตโนมัติโดยผู้ใช้ไม่รู้สึกอะไร

อาการเด็กเรียนอยู่แล้วโดนเด้งออก เกือบทั้งหมดเกิดจาก 3 จุดนี้ คือ access token หมดแล้วระบบต่ออายุไม่สำเร็จ, refresh token หมดอายุจริง หรือ logic ฝั่ง client จัดการตอนต่ออายุพลาด เช่น สอง request แย่งกันต่ออายุพร้อมกัน การไล่แก้ P0 เรื่อง session จึงต้องตรวจครบทั้งสามจุด ไม่ใช่แค่ขยายตัวเลขอายุ token อย่างเดียว

ปัญหาเชิงโครงสร้างของระบบปัจจุบัน

  • Login ซ้ำซ้อนข้ามเว็บ user เข้า online แล้ว มาถึง classroom ต้อง login ใหม่ เพราะ token ของสองเว็บออกคนละใบ ไม่รู้จักกัน
  • อายุ session สองฝั่งไม่เท่ากัน ฝั่งหนึ่งหลุดอีกฝั่งยังอยู่ user เลยงงว่าตกลงตัวเอง login อยู่หรือเปล่า
  • คนเดียวกลายเป็นสองบัญชี เช่น login ด้วย LINE ที่เว็บหนึ่ง Facebook ที่อีกเว็บ ระบบมองเป็นคนละคน ประวัติการเรียนกับการซื้อแยกจากกัน

ระยะสั้น ทำได้ใน 1 ถึง 2 sprint

  • ยืดอายุ refresh token ฝั่ง classroom ให้ยาวพอกับพฤติกรรมเรียนจริง เช่น 30 วันแบบ sliding คือใช้งานเมื่อไหร่นับหนึ่งใหม่ เด็กที่เข้าเรียนทุกสัปดาห์จะไม่หลุดเลย
  • ทำ silent refresh ให้แน่น client ต้องต่ออายุ token ก่อนหมดจริง และมี retry เมื่อการต่ออายุล้มเหลวชั่วคราว ไม่ใช่เด้ง logout ทันที จุดนี้มักเป็นต้นตอ session หลุดมากกว่าตัวเลข expire เอง
  • ใช้ cookie ระดับ parent domain ช่วยเชื่อม สองเว็บอยู่ใต้ smartmathpro.com เหมือนกัน ตั้ง cookie ที่ .smartmathpro.com ให้อ่านร่วมกันได้ แล้วทำ logic ว่าถ้ามี session ฝั่งโน้นอยู่ ให้ login ฝั่งนี้อัตโนมัติ เป็น SSO ฉบับประหยัดที่ไม่ต้องรื้อระบบ
  • ใช้เบอร์โทรเป็น key จับคู่บัญชีซ้ำ เด็กไทยหนึ่งคนมักมีเบอร์เดียว จึงเหมาะเป็นตัวกลางบอกว่า LINE account นี้กับ Facebook account นี้คือคนเดียวกัน บังคับยืนยันเบอร์ครั้งแรกที่ login ทุกช่องทาง เพื่อเตรียมข้อมูลไว้รวมบัญชีในระยะยาว

ระยะยาว Auth Service กลางตัวเดียว

หลักคิดคือ LINE กับ Facebook ต้องถูกลดบทบาทจากระบบบัญชี เหลือแค่วิธี login ส่วนบัญชีจริงมีใบเดียวอยู่ที่ระบบกลาง ทั้งสองเว็บเลิกออก token เอง รับจากระบบกลางอย่างเดียว นโยบายอายุ token จึงเป็นชุดเดียวกันโดยอัตโนมัติ

User (นักเรียน) | v เลือกช่องทาง login ช่องทางใดก็ได้ LINE / Facebook / เบอร์โทร+รหัสผ่าน (เป็นแค่วิธีพิสูจน์ตัวตน ไม่ใช่ตัวบัญชี) | v Auth Service กลาง จับคู่เข้าบัญชีเดียว 1 คน = 1 บัญชี ออก token จากที่เดียว นโยบาย expire ชุดเดียว | ┌──────────┴──────────┐ v v online classroom ใช้ token ใบเดียวกัน login ครั้งเดียวเข้าได้ทั้งสองเว็บ

ทางเลือกเชิงเครื่องมือมีตั้งแต่บริการสำเร็จรูปอย่าง Auth0, Clerk, Supabase Auth หรือ self-host อย่าง Keycloak สำหรับทีมขนาดเรา แนะนำของสำเร็จรูปก่อน เพราะงาน auth พลาดแล้วแพง และไม่ใช่ความสามารถหลักที่เราต้องสร้างเอง

นโยบาย expire ที่แนะนำเป็นจุดตั้งต้น

สำหรับ EdTech ที่ผู้ใช้เป็นนักเรียน ให้ access token อายุ 30 ถึง 60 นาที, refresh token 30 วันแบบ sliding และบังคับยืนยันตัวตนซ้ำเฉพาะจุดอ่อนไหวอย่างหน้าชำระเงินหรือเปลี่ยนรหัสผ่าน แนวนี้สมดุลระหว่างความปลอดภัยกับการไม่รบกวนการเรียน ตัวเลขปรับจูนภายหลังได้จากข้อมูล session หลุดจริง

เรื่องที่ต้องออกแบบเพิ่ม เพราะใช้เบอร์โทรกับรหัสผ่าน

ต่างจากระบบ OTP ตรงที่รหัสผ่านอยู่กับตัวผู้ใช้ตลอด ไม่มีตัวกลางอย่าง SMS มาช่วยยืนยันซ้ำ จึงมีสองเคสที่ต้องเตรียม flow ไว้ก่อน

  • ลืมรหัสผ่าน ถ้าบัญชีนั้นผูก LINE หรือ Facebook ไว้ด้วย ให้ reset ผ่านช่องทางนั้นได้เลย ถูกกว่าและเด็กทำเองได้โดยไม่ต้องรอ support และทั้ง flow ต้องจบใน classroom ได้ ไม่เด้งข้ามเว็บแบบปัจจุบัน
  • บัญชีที่มีแค่เบอร์กับรหัสผ่าน ไม่มี social ผูก ถ้าลืมรหัสผ่านแล้วเบอร์เปลี่ยนด้วย จะกู้บัญชีเองไม่ได้เลย ต้องมีเส้นทางกู้ผ่านแอดมินพร้อมวิธีพิสูจน์ตัวตน เช่น ประวัติการซื้อ วางแผนไว้ก่อนจะได้ไม่กลายเป็น ticket ก้อนใหญ่ของทีม support
  • เด็กเปลี่ยนเบอร์บ่อยกว่าผู้ใหญ่ ต้องมี flow เปลี่ยนเบอร์ที่ทำเองได้โดยยืนยันผ่าน social ที่ผูกไว้ และจุดนี้คือเหตุผลที่ควรชวน user ผูก social เพิ่มอย่างน้อยหนึ่งช่องทางตั้งแต่ onboarding
ข้อควรระวังตอน migrate บัญชีเข้าระบบกลางต้องมี flow จับคู่บัญชีซ้ำสำหรับคนที่มีบัญชีทั้งสองเว็บก่อนเปิดใช้จริง โดยใช้เบอร์โทรเป็น key หลัก และมีหน้ายืนยันให้ user ตรวจว่าใช่บัญชีตัวเองจริงก่อนรวม ห้ามรวมอัตโนมัติเงียบ ๆ เพราะถ้ารวมผิดคน ประวัติการเรียนและการซื้อจะปนกันและแก้ยากมาก
